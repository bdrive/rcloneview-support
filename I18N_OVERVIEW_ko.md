> 팀 공유용 문서. 무엇을 하고 있고, 어떤 문제를 만났고, 왜 배포 구조를 바꿔야 하는지 설명한다.
> 작업 절차·명령어 등 실무 상세는 [I18N_RUNBOOK_ko.md](./I18N_RUNBOOK_ko.md) 참고.

## 1. 무엇을 하는가

support 사이트(블로그 620개 + howto/tutorials 문서)를 **RcloneView Desktop과 동일한 9개 언어**로 제공한다.

| 로케일 | 언어 | | 로케일 | 언어 |
|---|---|---|---|---|
| `en` | 영어 (원문) | | `de` | 독일어 |
| `ko` | 한국어 | | `es` | 스페인어 |
| `ja` | 일본어 | | `fr` | 프랑스어 |
| `zh-Hans` | 중국어 간체 | | `id` | 인도네시아어 |
| `zh-Hant` | 중국어 번체 | | | |

- URL: `/support/`(영어), `/support/ko/...`, `/support/ja/...` 형태. hreflang 자동 생성으로 SEO 처리.
- 번역은 Claude(Sonnet) 파이프라인으로 생성하고, 4단계 자동 검증 후 커밋.
- **진행 현황(2026-07-09): 620개 중 360개 완료 (58%)**, 검증·전체 빌드 통과 상태.

## 2. Docusaurus i18n의 동작 방식 — 그리고 만난 문제

Docusaurus의 다국어는 **"언어 하나 = 사이트 하나"** 방식이다:

- 번역본은 언어별 **파일 복사본**으로 관리 (`i18n/ko/...`, `i18n/ja/...`)
- 빌드하면 **언어마다 사이트 전체를 다시 생성** — 이미지·JS까지 각 언어 폴더에 복사됨

이 구조라서 다음이 발생한다 (실측):

| | 영어만 (현재 운영) | 9개 언어 빌드 |
|---|---|---|
| 산출물 용량 | 337MB | **3.1GB** (이미지가 9벌 복제, 2GB 이상이 중복) |
| 파일 수 | 약 4,000개 | **36,126개** |
| 빌드 시간 | 1~2분 | **약 14분** (풀 빌드 9회) |

### 왜 이게 "배포" 문제인가

지금 배포는 **빌드 산출물을 www 저장소의 `support/` 폴더에 커밋**하고, main 머지 시 Cloudflare가 www를 통째로 반영하는 방식이다. 9개 언어가 되면:

1. **Cloudflare Pages는 배포당 20,000파일 제한** — 36,000파일을 커밋해 머지하는 순간 배포가 실패한다. 즉 현재 방식으로는 다국어 빌드를 반영할 방법이 없다.
2. www 저장소가 지금도 1.5GB(대부분 support 산출물 이력)인데, 매일 블로그 봇 커밋마다 9개 언어의 sitemap/RSS/페이지네이션이 전부 바뀌어 이력이 계속 불어난다.
3. PR에 빌드 산출물 수천 파일이 섞여 diff 리뷰가 사실상 불가능하다.

## 3. 검토한 대안: Astro로 갈아타면? (NetDrive 블로그처럼)

bdweb의 NetDrive 블로그(Astro 5)를 심층 분석했다. 결론: **부분적으로만 해결되고, 지금 갈아타는 건 손해.**

| 문제 | Astro로 옮기면 |
|---|---|
| 산출물 ×9 (이미지 복제) | ✅ 구조적으로 해결 — 단일 빌드, 이미지 한 벌 공유 (추산 ~600MB / ~8,000파일) |
| 원문-번역 동기화 | ❌ 동일 — Astro도 언어별 파일 복사본. 번역 파이프라인이 해결할 문제 |
| i18n 편의 기능 | ❌ 전부 수제작 — 미번역 글 영어 fallback, hreflang, 언어별 RSS/sitemap, 태그, 문서 사이드바를 직접 구현해야 함 (Docusaurus는 기본 제공, 이미 구축 완료) |

참고 사실:
- NetDrive 블로그는 **영어 단독**(97개 글)이다. i18n은 헤더/푸터 라벨 en/ko 토글뿐 — "Astro 다국어 사례"가 아니어서 가져다 쓸 코드가 없다.
- 우리 콘텐츠 3,500여 파일에는 Docusaurus 전용 문법(`:::tip`, `<!-- truncate -->`, `@site/src` React import)이 박혀 있어 이관 시 전량 변환이 필요하다.
- 기존 URL 620개×9를 깨지 않고 옮겨야 하는 SEO 리스크도 있다.

**판단**: 산출물 크기 하나를 해결하려고 나머지 전부를 다시 만들 가치가 없다. 산출물 문제는 아래처럼 배포 변경만으로 해결된다. (단, 신규 사이트를 오늘 처음 만든다면 Astro도 합리적 선택지다.)

## 4. 제안하는 구조: 빌드 산출물을 git 밖으로

> **[결정 2026-07-18] 이 장의 제안은 철회됨.** 계정이 Pages **Pro 플랜(파일
> 10만 개 한도)**으로 확인됐고 (아래 도식의 "20k 제한"은 Free 플랜 기준),
> 빌드도 846MB/약 33k 파일로 감축돼 한도 내이므로 **기존 방식(산출물을 www에
> 커밋)을 유지**하기로 했다. 구현했던 R2 킷은 저장소에서 제거(git 히스토리에
> 보존). 아래 내용은 검토 기록으로 남긴다.

**"support 산출물을 www 저장소에 커밋"을 중단**하고, CI가 빌드해서 R2 오브젝트 스토리지에 올리고, Worker가 서빙한다. NetDrive 블로그가 이미 같은 패턴(GitHub Actions 빌드 → nginx 서버 rsync, `bdweb/.github/workflows/blog-deploy.yml`)으로 운영 중이라 팀에 검증된 방식이다.

```
[변경 전]
글 작성 → 빌드 → 산출물을 www에 커밋 → PR → 머지 → Cloudflare 반영
                  (3.1GB/36k 파일이 git에 들어감 → Pages 20k 제한에 걸려 불가)

[변경 후]
글 작성(소스 md 9개) → support 저장소에 PR → 리뷰·머지
                                              ↓ (GitHub Actions 자동)
                                    빌드(14분) → rclone sync로 R2 업로드
                                              ↓
방문자 rcloneview.com/support/* ←— Cloudflare Worker가 R2에서 서빙
방문자 rcloneview.com/(나머지)  ←— www 기존 방식 그대로 (변화 없음)
```

### 구성 요소와 역할

| 구성 요소 | 역할 | 비고 |
|---|---|---|
| GitHub Actions | main 머지 감지 → 빌드 → R2 업로드. PR 시점엔 빌드 검사만 | bdweb blog-deploy.yml과 동일 패턴 |
| R2 버킷 | 빌드 산출물 저장소 (3.1GB, 파일 수 제한 없음) | `rclone sync`로 증분 업로드 |
| Cloudflare Worker | `/support/*` 요청을 R2에서 꺼내 서빙 (~50줄) | 나머지 경로는 기존 www로 통과 |
| www 저장소 | `support/` 폴더 제거, 홈페이지 전용으로 회귀 | 배포 방식 변화 없음 |

### 팀원 입장에서 바뀌는 것 / 안 바뀌는 것

- **안 바뀜**: PR 확인 → 머지 흐름, 사이트 URL, 방문자 화면 전부 그대로.
- **좋아짐**: PR이 마크다운 소스만 담게 되어 리뷰 가능해짐 (글 1개 = 원문 1 + 번역 8 = 9파일). PR에 빌드 통과 ✅/❌가 미리 표시됨.
- **비용**: R2 무료 구간(저장 10GB, egress 무제한 무료) 안. Worker는 무료 플랜 10만 요청/일, 초과 시 월 $5. 사실상 0~$5/월.

### 사람이 한 번만 해야 하는 초기 설정

1. Cloudflare 대시보드에서 R2 버킷 생성 + API 토큰 발급
2. GitHub support 저장소 Settings → Secrets에 토큰 등록 (`R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_ENDPOINT`)
3. Worker 배포 승인 (`wrangler deploy` 시 Cloudflare 인증)

이후는 전부 자동.

## 5. 남은 작업 (2026-07-09 기준)

1. 소급 번역 완료 — 260개 남음 (세션 윈도우 3회분)
2. 신규 글 봇에 번역 단계 통합 — 글 1개 발행 시 8개 번역 동시 생성
3. 배포 전환 — 위 4장 구조 구축 (R2/Worker/Actions), www에서 support 제거
4. 배포 시 언어별 sitemap을 Google Search Console/Bing에 등록
5. (별건) 홈페이지 rcloneview_www 자체의 다국어화 — 정적 HTML이라 별도 설계 필요, Astro가 후보

### SEO 참고

- hreflang이 언어별 페이지 관계를 선언하므로 중복 콘텐츠 문제는 없다.
- 유일한 리스크는 **부분 번역 상태의 장기화** — 미번역 글은 로케일 URL에 영어가 그대로 노출(fallback)되므로, 소급 번역을 빨리 끝내는 것이 좋다.
